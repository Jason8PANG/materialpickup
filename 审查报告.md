## 二、严重问题（必须修复）

### S-1. SQL 注入风险（f-string 拼接 SQL WHERE 子句）

**影响文件**：`app/routes/request_bp.py`、`app/routes/warehouse.py`、`app/routes/admin.py`

多个接口使用 f-string 动态构造 SQL WHERE 子句，例如 `request_bp.py:123,129,133`:
```python
where_clause = " AND ".join(where)                          # where 条件来自外部拼接
cursor.execute(f"SELECT COUNT(*) as total FROM kr_material_request WHERE {where_clause}", params)
```

虽然 `params` 使用了参数化绑定，但 `where` 列表中的条件本身是可控字符串（如 `"status = %s"`），当未来添加新筛选条件时容易引入注入点。**这是代码坏味道，应在迭代中彻底消除。**

**修复建议**：使用 SQLAlchemy ORM 或构建安全的动态查询构建器。

### S-2. 硬编码敏感信息

**影响文件**：`app/config.py`

| 行号 | 配置项 | 问题 |
|:----:|--------|------|
| 5 | `SECRET_KEY` | `'material-kanban-secret-key-2024'` — Session 签名密钥硬编码，可被暴力破解伪造 Session |
| 11 | `MYSQL_PASSWORD` | `'!Q1234567'` — 数据库密码明码硬编码在代码库中 |
| 19 | `LOCAL_PASSWORD` | `'kanban2024'` — 所有用户共享本地密码，无安全性可言 |

**修复建议**：
1. `SECRET_KEY` 应从环境变量读取，且使用 `secrets.token_hex(32)` 生成
2. 数据库密码应通过环境变量或密钥管理服务注入
3. 移除 `LOCAL_PASSWORD` 回退机制，或将其替换为每个用户的独立密码

### S-3. CSRF 防护缺失

**影响文件**：全局

所有 POST/PUT/DELETE 接口均未实施 CSRF 防护。攻击者可构造恶意页面，在用户已登录状态下跨站提交操作（审批、驳回、删除映射等）。

**修复建议**：使用 Flask-WTF 扩展添加 CSRF token，或对关键操作实施 Referer/Origin 校验。

### S-4. Session 配置不安全

**影响文件**：`app/config.py` + `app/routes/auth.py`

| 问题 | 说明 |
|------|------|
| 缺少 `SESSION_TYPE` | Flask 默认使用客户端 Session（Cookie），内容可被解码（虽经签名但只防篡改不防窥探） |
| 缺少 `PERMANENT_SESSION_LIFETIME` | `session.permanent = True` 已设置，但未配置过期时长 |
| 缺少 `SESSION_COOKIE_HTTPONLY` | 未显式设置 HttpOnly 属性 |
| 缺少 `SESSION_COOKIE_SAMESITE` | 未限制跨站请求携带 Cookie |

**修复建议**：
```python
SESSION_TYPE = 'filesystem'               # 服务端 Session
SESSION_COOKIE_HTTPONLY = True
SESSION_COOKIE_SAMESITE = 'Lax'
SESSION_COOKIE_SECURE = True              # 生产环境
PERMANENT_SESSION_LIFETIME = 28800        # 8 小时
```

### S-5. 数据库连接泄漏风险

**影响文件**：所有 route 文件

每个请求都创建新的数据库连接（`get_db()`），且部分代码使用裸 `try...finally` 而非上下文管理器。如果在 `cursor.execute()` 和 `db.close()` 之间抛出异常，连接将泄漏，最终耗尽数据库连接池。

**修复建议**：
```python
from contextlib import contextmanager

@contextmanager
def get_db_connection():
    db = get_db()
    try:
        yield db
    finally:
        db.close()
```

### S-6. 本地密码绕过认证

**影响文件**：`app/routes/auth.py:23-26`

```python
if not ldap_user:
    # 本地密码验证
    if password != Config.LOCAL_PASSWORD:
        return jsonify({'success': False, 'message': '用户名或密码错误'}), 401
```

任何人只要知道 `LOCAL_PASSWORD`（当前为 `kanban2024`）即可登录任意已知用户账号。

**修复建议**：移除本地密码回退机制，或改为每个用户独立的密码存储（hash+盐）。

## 三、建议改进

### I-1. LDAP 认证未实现（功能缺失）

`app/services/ldap_service.py:26` — `ldap_authenticate()` 始终返回 `None`，实际从未调用 LDAP。根据需求文档 F-001，LDAP 是 P0 功能。

**修复建议**：使用 `python-ldap` 或 `ldap3` 库实现 LDAP 认证；若暂时不需要 LDAP，应移除该模块和回退逻辑，改用独立密码系统。

### I-2. 审批邮件从未发送（功能缺失）

`app/services/email_service.py` 中 `send_approval_email()` 函数完整，但**没有任何代码在申请提交后调用它**。需求 F-005/F-012 规定申请提交后自动发送审批邮件。

**修复建议**：在 `request_bp.py` 的 `create_request()` 中，提交成功后查询主管邮箱并调用 `send_approval_email()`。

### I-3. 数字类型转换未处理异常

**影响文件**：`app/routes/request_bp.py:34-37`、`warehouse.py:227`、`admin.py:153-154`

```python
float(data['quantity'])         # 若 data['quantity'] 为无效数值字符串将抛出 ValueError
int(request.args.get('page', 1))  # 若 page 参数不可转换将抛出 ValueError
```

**修复建议**：使用 `try/except` 包裹类型转换，或在获取参数时加校验。

### I-4. 重复代码过多

| 重复模式 | 出现位置 | 建议 |
|----------|----------|------|
| `STATUS_LABELS` 字典 | `kanban.py`、`approval.py`、`warehouse.py`、`request_bp.py` | 抽取到公共模块 |
| 分页查询 + 计数模式 | `warehouse.py`、`request_bp.py`、`admin.py` 共 5 处 | 抽取通用分页函数 |
| `add_log()` 函数 | `approval.py` 和 `warehouse.py` 各定义一份 | 抽取到公共模块 |
| `check_admin()` 模式 | `admin.py` 独自定义 | 抽取到公共装饰器或 middleware |

### I-5. 缺少全局错误处理器

当前没有任何 `@app.errorhandler` 或 `@app.before_request` 统一处理逻辑。若路由内抛出未捕获异常，Flask 将返回默认的 500 Internal Server Error 或暴露 debug 信息（因 `debug=True`）。

**修复建议**：在 `create_app()` 中添加：
```python
@app.errorhandler(404)
def not_found(e):
    return jsonify({'success': False, 'message': '资源不存在'}), 404

@app.errorhandler(500)
def server_error(e):
    return jsonify({'success': False, 'message': '服务器内部错误'}), 500
```

### I-6. API 路径不符合文档约定

需求文档规定 API 路径前缀为 `/api/v1/`，但实际路由均为 `/api/` 开头。这会导致前后端对接时路径不一致。

**修复建议**：在注册蓝图时添加 `url_prefix`：
```python
app.register_blueprint(auth_bp, url_prefix='/api/v1/auth')
app.register_blueprint(kanban_bp, url_prefix='/api/v1/kanban')
```

### I-7. debug 模式暴露敏感信息

`run.py:28` — `app.run(host='0.0.0.0', port=5000, debug=True)`

生产环境中 `debug=True` 会在错误页面展示完整的调用栈和本地变量。同时绑定 `0.0.0.0` 暴露到所有网络接口。

**修复建议**：
```python
if __name__ == '__main__':
    debug_mode = os.environ.get('FLASK_DEBUG', 'false').lower() == 'true'
    app.run(host='0.0.0.0', port=5000, debug=debug_mode)
```

### I-8. 响应式设计未完全实现

CSS 中移动端 `.kanban-board` 使用 `flex-direction: column`，但断点未按需求文档（PC: 5列, 平板: 2-3列, 手机: 1列）实现。平板断点（768-991px）被忽略。

### I-9. utils/__init__.py 空文件

`app/utils/__init__.py` 完全为空，未包含任何工具函数。代码中实际需要但没有实现的工具包括：公共分页函数、统一 JSON 响应封装、权限装饰器等。

### I-10. Jinja2 模板中内联 JS 过大

看板页面（`kanban.html`、`request_form.html`、`request_detail.html`、`pending_list.html`、`history.html`、`admin_mappings.html`）均包含大量内联 JS。应将业务 JS 抽离到独立的 `.js` 文件中。
